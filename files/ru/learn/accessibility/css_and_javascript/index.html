CSS и JavaScript имеют обширные возможности для создания доступных итерфейсов. Но это обоюдоострый клинок — при неосторожном обращении они могут доступности и значительно навредить. В данной статье мы хотели бы рассмотреть некоторые лучшие практики в CSS и JavaScript, которые позволят даже сложному контету оставался доступным настолько, насколько это возможно.

Необходимые знания: Базовая компьютерная грамотность, базовое понимание HTML, CSS и JavaScript, а также понимание что такое доступность.
Цель: Ознакомиться со способами использования CSS и JavaScript таким образом, чтобы добиваться наилучшей возможной доступности.

Доступны ли CSS и JavaScript?

В сравнении с HTML, вклад CSS и JavaScript в доступность не настолько велик, однако, в зависимости от использования, они всё равно способны на неё повлиять. Иными словами, важно изучить некоторые рекомендации, чтобы CSS и JavaScript ненароком не ломали доступность ваших веб-страниц.

CSS

Начнем с CSS.

Правильная семантика и ожидаемое поведение

С помощью CSS возможно заставить любой HTML-элемент выглядеть как угодно, однако это не означает, что делать так — правильно. Как мы часто упоминаем в статье HTML: Хорошая основа для доступности, для каждой задачи всегда следует использовать соответствующий семантический элемент. Если не следовать данному правилу, то могут возникать проблемы и неудобства для всех, а в особенности для людей с ограниченными возможностями. Правильное использование семантики тесно связано с ожидаемым пользователем поведением — элементы выглядят и ведут себя определенным образом, в соответствии с их функциональностью.

Например, пользователь скринридера не сможет передвигаться по странице с помощью заголовков, если разработчик не разметит контент с помощью заголовков надлежащим образом. Аналогично, если заголовок стилизован так, что выглядит не как заголовок, то он теряет свое визуальное значение.

Эмпирическое правило звучит следующим образом: стилизовать элементы страницы, подгоняя их по ваш дизайн, можно до той степени, пока они выглядят и ведут себя ожидаемым образом. В следующих разделах мы рассмотрим главные возможности HTML.

"Стандартная" структура текстового контента

Заголовки, параграфы и списки — это основа текстового контента веб-страницы:

<h1>Заголовок</h1>

<p>Параграф</p>

<ul>
  <li>В моем списке</li>
  <li>два элемента.</li>
</ul>

Типичный CSS может выглядеть следующим образом:

h1 {
  font-size: 5rem;
}

p, li {
  line-height: 1.5;
  font-size: 1.6rem;
}

Нам следует убедиться, что:

  • Выбраны разумные размер шрифта, интерлиньяж, межбуквенное расстояние и прочее, чтобы сделать текст логичным, разборчивым и комфортным для чтения.
  • Заголовки выделяются на фоне основного текста, например имеют такой же крупный размер шрифта и полужирное начертание, как стандартная стилизация. Списки выглядят как списки.
  • Цвет текста должен быть достаточно контрастен относительно цвета фона.

 Для более подробной информации смотрите Основы редактирования текста в HTML и Стилизация текста.

Акцентированный текст

Строчный элемент, который акцентирует особое внимание на содержащемся в нем тексте:

<p>Вода <em>очень горячая</em>.</p>

<p>Собирающиеся на поверхности капельки воды называются <strong>конденсатом</strong>.</p>

Возможно нам понадобится добавить цвета акцентированному тексту

strong, em {
  color: #a60000;
}

Однако, не стоит сильно менять стандартную стилизацию акцентирующих элементов. Стандартные паттерны полужирного и курсивного текста хорошо узнаваемы, поэтому изменения стилизации может вызвать путаницу. Для более подробной информации смотрите Акцент и важность.

Аббревиатуры

Элемент, который разметить аббревиатуру или акроним и указать их расшифку:

<p>Веб-контент верстается с помощью <abbr title="Hypertext Markup Language">HTML</abbr>.</p>

И вновь, возможно нам понадобится немного стилизовать наши аббревиатуры:

abbr {
  color: #a60000;
}

Стандартный паттерн стилизации для аббревиатур — это пунктирное подчеркивание, не стоит от этого сильно отклоняться. Для более подробной информации смотрите Аббревиатуры.

Ссылки

Гиперссылки помогают нам переходить из одного места в сети в другое:

<p>Посетите <a href="https://www.mozilla.org">домашнюю страницу Mozilla</a>.</p>

Ниже приведена базовая стилизация для ссылок:

a {
  color: #ff0000;
}

a:hover, a:visited, a:focus {
  color: #a60000;
  text-decoration: none;
}

a:active {
  color: #000000;
  background-color: #a60000;
}

Стандартная стилизация ссылки — подчеркивание и отличный от стандартного цвет (по-умолчанию, синий) в обычном состоянии, иной цвет в случае, если ссылка ранее была посещена (по-умолчанию, фиолетовый) и еще один цвет для состояния нажатости (по-умолчанию, красный). Кроме того, при наведении на ссылку указателя мыши, его иконка изменяется на указующую руку, а также ссылка выделяется когда находится в фокусе (при перемещении по странице с помощью клавиши tab) или активна. Изображение ниже показывает выделение в Firefox (пунктирная обводка) и Chrome (синяя обводка):

До тех пор, пока пользователи получают обратную связь при взаимодействии со ссылками, можно проявлять определенную креативность относительно их стилизации. Что-нибудь обязательно должно происходить при изменении состояния. Не следует также отключать стандартную иконку указателя мыши или обводку — она вносит очень важный вклад в доступность для тех, кто использует клавитуру для навигации по сайту.

Формы

Формы - это DOM-элементы, которые позволяют пользователям вводить данные на веб-сайтах:

<div>
  <label for="name">Введите имя</label>
  <input type="text" id="name" name="name">
</div>

Вы можете найти пример хорошей стилизации формы здесь: form-css.html (а также посмотреть вживую).

При стилизации форм обычно производится подгонка размеров элементов, выравнивание лейблов и полей ввода, а также придание им изящного внешнего вида.

Не следует сильно отклоняться от стандартной стилизации фокуса на элементах формы. По-умолчанию, это состояние выглядит так же, как у ссылок (смотрите выше). Стилизовать фокус и наведение указателя мыши на элементы формы можно, например, для придания им универсального внешнего вида в разных браузерах или же для лучшего соответствия дизайну сайта. При этом не стоит совсем отключать отображение этих состояний, потому что с их помощью пользователи ориентируются в происходящем на странице.

Таблицы

Таблицы используются для отображения табличных данных.

Вы можете найти хороший пример HTML и CSS для таблицы в нашем примере: table-css.html (а также посмотреть вживую).

Стилизуют таблицы обычно для придания им более симпатичного внешнего вида и лучшего попадания в общий дизайн страницы. Не стоит отказываться от выделения заголовков (обычно с помощью полужирного начертания) и чередования двух цветов строк для лучшего восприятия информации.

Цвета и контраст

При выборе цветовой палитры для сайта, следует следить за тем, чтобы цвет текста имел высокий уровень контрастности относительно цвета фона. Дизайн может выглядеть очень стильно, однако это не будет иметь никакого смысла, если люди с дефектами зрения (такими как дальтонизм) не смогут прочитать ваш контент.

К счастью, проверить все ли в порядке с уровнем контрастности весьма несложно. Существует множество онлайн-инструментов, которые помогут сравнить пару цветов и вычислить значение контрастности. Например, Color Contrast Checker от WebAIM. Это довольно простой в использовании инструмент, который также предоставляет инструкции как вам добиться соответствия критериям WCAG по контрастности цветов.

Примечание: Высокий уровень контрастности также позволяет пользователям мобильных устройств с глянцевыми экранами комфортнее читать текст при ярком освещении.

Кроме того, не следует полагаться только на цвет при выделении важной информации, потому что некоторые пользователи не распознают цвета. Например, стоит не просто пометить обязательные поля формы красным, но также добавить звёздочку рядом.

Скрытые элементы

Существует множество ситуаций в дизайне, когда не нужно показывать сразу весь контент. Например, в нашем прототипе Панели вкладок (здесь можно посмотреть исходный код) имеется три секции с информацией, но мы располагаем их одна над другой и предоставляем набор заголовков вкладок, которые позволяют переключаться между вкладками при клике (они также достпны с помощью клавитуры — и можно выбирать используя клавиши Таб и Enter/Return).

В целом это никак не касается пользователей скринридеров — до тех пор, пока контент в верстке расположен в правильном порядке, у них не будет никаких проблем. Абсолютное позиционирование (как в примере выше) считается одним из лучших механизмов для визуального скрытия контента, потому что никак не мешает скринридерам.

А вот {{cssxref("visibility")}}:hidden и {{cssxref("display")}}:none, напротив, использовать не стоит, потому что они, в свою очередь, контент от скринридеров успешно скрывают. Конечно, только если это не является вашей целью.

Примечание: В статье Invisible Content Just for Screen Reader Users содержится множество полезной информации по данной теме.

Помните, что пользователи могут переопределять ваши стили

У пользователей есть возможность переопределять ваши стили своими, например:

  • В руководстве Сары Мэддокс Как использовать пользовательские таблицы стилей (CSS) в Firefox (EN) можно найти удобные инструкции по пользовательским стилям в Firefox, а в статье Как использовать пользовательские таблицы стилей (CSS) в Internet Explorer за авторством Адриана Гордона, соответственно, для IE.
  • Также можно воспользоваться браузерными расширениями. Например, расширение Stylish доступно для Firefox, Safari, Opera и Chrome.

Пользователи могут делать это по множеству причин. Пользователям со слабым зрением может понадобиться сделать текст крупнее на всех сайтах, которыми они пользуются. А пользователю с монохромазией потребуется перекрасить все сайты высококонтрастные цвета, которые ему проще различать. Какая бы причина ни была, следует делать ваш дизайн достаточно гибким, чтобы подобные изменения в нем работали. Например, стоит убедиться, что область с главным контентом будет нормально обрабатывать более крупные размеры шрифта — она не должна просто обрезать текст или совсем ломаться. Вполне нормально, если просто появятся полосы прокрутки.

JavaScript

При неправильном использовании, JavaScript тоже способен вредить доступности.

Современный JavaScript — это очень мощный язык, с помощью которого можно делать кучу разных вещей: от простых обновлений пользовательского интерфейса до полноценных 2D и 3D игр. Не существует строгого правила, что любой тип контента должен быть на 100% доступен для всех — просто нужно прилагать все усилия, чтобы делать веб-приложения доступными настолько, насколько это возможно.

Простой контент сравнительно несложно сделать доступным — например, текст, изображения, таблицы, формы и кнопки. Как было разобрано в статье HTML: Хорошая основа для доступности, основные тезисы следующие:

  • Хорошая семантика: использование каждого элемента для своей задачи. Например, следует убедиться, что используются различные типы элементов: заголовки, параграфы, {{htmlelement("button")}}, {{htmlelement("a")}} и так далее.
  • Контент должен быть доступен либо как обычный текст, либо как альтернативный текст, например атрибут alt для изображений.

Мы также показали пример использования JavaScript для добавления некоторым элементам свойств, которых по-умолчанию они лишены — смотрите в разделе Добавление доступности с клавиатуры. Такую возможность стоит рассматривать только для ситуаций, когда, по какой-то причине, у вас нет возможности изменять разметку. Намного надежнее и универсальнее просто использовать для каждой задачи соответствующий элемент. Другой способ улучшить доступность для не семантичных JavaScript-виджетов — это использование WAI-ARIA для добавления некоторой искусственной семантики для пользователей скринридеров. В следующей статье мы рассмотрим эту тему подробнее.

Сложные приложения, например 3D игры, сделать доступными непросто — написанная с использованием WebGL игра рендерится в элементе {{htmlelement("canvas")}}, который в настоящее время не предоставляет никаких возможностей для использования людьми с дефектами зрения. Можно оправдываться тем, что данная группа не входит в целевую аудиторию, поэтому не стоит ожидать полной доступности. Однако вы всё ещё моежете реализовать управление с помощью клавиатуры для игроков, которые не могут использовать мышь и сделать цветовую схему достаточно контрастной для игроков с дальтонизмом.

JavaScript головного мозга

Данная проблема возникает, когда разработчики полагаются на JavaScript слишком сильно. Порой, вы можете встретить сайт, на котором все сделано с помощью JavaScript — HTML, CSS и так далее. Такое решение точно так же требует всех видов доступности и проблем с ней связанных, поэтому не рекомендуется.

Для каждой задачи хорош свой инструмент — это правило справедливо как для HTML-элементов, так и для используемой технологии в целом. Стоит хорошенько обдумать, нужно ли использовать тот восхитительный, написанный на JavaScript 3D блок с информацией или же воспользоваться старым добрым текстовым блоком. Нужен ли тот сложный нестандартный виджет в форме или будет достаточно простого поля для ввода текста. Не следует генерировать весь HTML с помощью JavaScript, если это возможно.

Ненавязчивый JavaScript

Когда вы создаете свой контент, стоит помнить о понятии ненавязчивый JavaScript. Идея ненавязчивого JavaScript состоит в том, что следует использовать его только для дополнительного улучшения пользовательского опыта — базовые возможности, в идеале, должны работать без JavaScript. И вновь, здесь мы говорим в основном об использовании стандартных браузерных возможностей когда это возможно.

Хорошие примеры использования ненавязчивого JavaScript:

  • Валидация формы на клиенте, которая показывает пользователю сообщение о проблемах с заполнением формы без необходимости отправлять данные для проверки на сервер. Если скриптов не будет, то форма продолжит работить, просто валидация будет несколько медленнее.
  • Создание нестандартных элементов управления для HTML5 <video>, которыми возможно пользоваться с клавиатуры. И в то же самое время, указание прямой ссылки на видео, которой можно воспользоваться, если JavaScript недоступен (стандартные элементы управления <video> не доступны с клавиатуры в большинстве браузеров).

Для примера, мы подготовили простенькую черновую версию клиентской валидации формы — form-validation.html (здесь можно посмотреть вживую). Если попытаться нажать на кнопку отправки не заполнив одно из полей, то отправка не произойдет и появится сообщение, с информацией что пошло не так.

Такой тип валидации формы является ненавязчивым — вы все еще сможете абсолютно корректно использовать форму вообще без JavaScript. Ведь любая форма для ввода важной информации будет также валидироваться и на сервере — потому что злонамеренные пользователи могут очень просто обойти клиентскую валидацию (например, просто отключив JavaScript в браузере). При этом валидация на стороне клиента всё ещё остается очень польезной для показа ошибок — пользователи мгновенно узнают об ошибках в вводе, вместо того, чтобы ждать цикла: отправка на сервер - валидация - перезагрузка страницы. Это очень хорошее улучшение пользовательского опыта.

Примечание: В этом простом примере серверная валидация не реализована.

Также мы постарались сделать эту валидацию формы довольно доступной. Мы использовали элементы {{htmlelement("label")}} и корректно связали их со своими полями ввода, таким образом скринридеры смогут корректно их озвучить:

<label for="name">Введите ваше имя:</label>
<input type="text" name="name" id="name">

Мы производим валидацию только при отправке формы, чтобы избежать слишком частого обновления пользовательского интерфейса и возможных проблем для пользователей скринридеров:

form.onsubmit = validate;

function validate(e) {
  errorList.innerHTML = '';
  for (let i = 0; i < formItems.length; i++) {
    const testItem = formItems[i];
    if (testItem.input.value === '') {
      errorField.style.left = '360px';
      createLink(testItem);
    }
  }

  if (errorList.innerHTML !== '') {
    e.preventDefault();
  }
}

Примечание: В данном примере мы скрываем и показываем сообщение об ошибке с именно помощью абсолютного позиционирования, а не каким-то иным методом, потому что такой способ никак не мешает скринридерам читать контент.

В реальности валидация форма может быть в разы сложнее — может понадобиться проверять, что введенное имя действительно выглядит как имя, а возраст — число, да ещё и реалистичное (например, не отрицательное и состоящее не более чем из 4 цифр). Здесь мы реализовали лишь простою проверку на наличие хоть какого-то текста в полях ввода (if (testItem.input.value === '')).

Когда валидация проходит успешно — форма отправляется на сервер. Если же были найдены ошибки (if (errorList.innerHTML !== '')), то мы прерываем отправку формы (при помощи preventDefault()) и отображаем сообщения обо всех найденых ошибках (смотрите далее). Следовательно, сообщения об ошибках будут показаны только если ошибки есть.

Для каждого пустого на момент отправки формы поля ввода мы создаем элемент списка с ссылкой и добавляем в errorList.

function createLink(testItem) {
  const listItem = document.createElement('li');
  const anchor = document.createElement('a');

  anchor.textContent = 'Поле ввода ' + testItem.input.name + ' пусто: введите ' + testItem.input.name + '.';
  anchor.href = '#' + testItem.input.name;
  anchor.onclick = function() {
    testItem.input.focus();
  };
  listItem.appendChild(anchor);
  errorList.appendChild(listItem);
}

Каждая ссылка служит двум целям — она рассказыает какая случилась ошибка, а также на неё можно кликнуть, чтобы перейти прямо в связанное поле ввода и скорректировать значение.

Примечание: Часть этого примера, связанная с focus(), довольно хитрая. Chrome и Edge (а также новые версии IE) переведут фокус на элемент, когда случится клик по ссылке, без необходимости в блоке кода onclick/focus(). Safari лишь выделит нужный элемент, так что этот блок кода нужен для установки фокуса. В Firefox же трюк с выделением элемента при клике по ссылке не работает вообще. Данная ошибка должна быть вскоре исправлена — тогда Firefox придет к паритету с прочими браузерами (смотрите {{bug(277178)}}).

Дополнительно, то что errorField расположено в самом начале исходного кода (тогда как в пользовательском интерфейсе при помощи CSS расположено иначе), позволит пользователям увидеть, что именно не так с заполнением формы и вернуться обратно к полям формы в начале страницы.

Кроме того, мы использовали несколько WAI-ARIA атрибутов в нашем примере, чтобы решить проблемы с доступностью, вызванные обновлением контента без перезагрузки страницы (по-умолчанию, скринридеры не могут подхватить эти изменения или уведомить пользователя):

<div class="errors" role="alert" aria-relevant="all">
  <ul>
  </ul>
</div>

Мы разберем эти атрибуты в следующей статье, которая подробно описывает WAI-ARIA.

Примечание: Кто-то из читателей мог задуматься о том факте, что в HTML5 формах есть встроенные механизмы валидации, такие как атрибуты required, min/minlength и max/maxlength (смотрите подробнее на странице элемента {{htmlelement("input")}}). Мы не использовали их в примере, потому что поддержка браузерами у HTML5 валидации варьируется (например, только IE10 и новее).

Примечание: Статья Usable and Accessible Form Validation and Error Recovery от WebAIM содержит дополнительную полезную информацию о доступной валидации форм.

Other JavaScript accessibility concerns

There are other things to be aware of when implementing JavaScript and thinking about accessibility. We will add more as we find them.

mouse-specific events

As you will be aware, most user interactions are implemented in client-side JavaScript using event handlers, which allow us to run functions in response to certain events happening. Some events can have accessibility issues. The main example you'll come across is mouse-specific events like mouseover, mouseout, dblclick, etc. Functionality that runs in response to these events will not be accessible using other mechanisms, like keyboard controls.

To mitigate such problems, you should double up these events with similar events that can be activated by other means (so-called device-independent event handlers) — focus and blur would provide accessibility for keyboard users.

Let's look at an example that highlights when this could be useful. Maybe we want to provide a thumbnail image that shows a larger version of the image when it is moused over or focused (like you'd see on an e-commerce product catalog.)

We've made a very simple example, which you can find at mouse-and-keyboard-events.html (see also the source code). The code features two functions that show and hide the zoomed-in image; these are run by the following lines that set them as event handlers:

imgThumb.onmouseover = showImg;
imgThumb.onmouseout = hideImg;

imgThumb.onfocus = showImg;
imgThumb.onblur = hideImg;

The first two lines run the functions when the mouse pointer hovers over and stops hovering over the thumbnail, respectively. This won't allow us to access the zoomed view by keyboard though — to allow that, we've included the last two lines, which run the functions when the image is focused and blurred (when focus stops). This can be done by tabbing over the image, because we've included tabindex="0" on it.

The click event is interesting — it sounds mouse-dependent, but most browsers will activate onclick event handlers after Enter/Return is pressed on a link or form element that has focus, or when such an element is tapped on a touchscreen device. This doesn't work by default however when you allow a non-default-focusable event to have focus using tabindex — in such cases you need to detect specifically when that exact key is pressed (see Building keyboard accessibility back in).
